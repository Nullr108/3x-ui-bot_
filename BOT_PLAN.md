# План Telegram-бота (3x-ui продажник + напоминалка + приём оплаты)

> Этот файл — описание того, КАК работает бот (workflow + состояния + тайминги).
> Какие методы API из 3x-ui нужны на каждом шаге — указано в блоках `API:`.
> Технический справочник по самому API — в [3X-UI_API_CHEATSHEET.md](3X-UI_API_CHEATSHEET.md).

---

## 0. Базовые соглашения и константы

| Параметр | Значение | Где используется |
|----------|----------|------------------|
| `email` клиента | `@<tg_username>` либо `temp-<tg_user_id>` (если username нет) | ключ клиента в 3x-ui |
| `tg_id` клиента | Telegram user_id | привязка, поиск, уведомления |
| Триал: срок | 1 день (сегодня X → ключ умирает X+1) | `expiry_time` |
| Триал: устройства | 2 | `limit_ip = 2` |
| Триал: трафик | безлимит (`total_gb = 0`) — ограничение только по времени | `total_gb` |
| Триал: инбаунды | ВСЕ инбаунды панели | цикл по `inbound.get_list()` |
| Цена подписки | 200 ₽ | статическая ссылка на оплату |
| Платная подписка: срок | 30 дней (месяц) | `expiry_time` |
| Проверка трафика после выдачи | через 4 часа | планировщик |
| Напоминание «протестируй» | за 3 часа до конца ключа | планировщик |
| Уведомление об окончании платной | за 1 день до конца | планировщик |

**КЛЮЧЕВОЕ ПРАВИЛО ОПЛАТЫ:**
оплата доступна **только** если у пользователя установлен Telegram-username
(он же `email` в 3x-ui в формате `@username`). Пока username = `temp-<id>`,
кнопка «Оплатить» сначала уводит на установку username, а не на оплату.

**Хранение состояния бота (локальная БД, напр. SQLite):**
для каждого пользователя надо хранить то, чего нет в 3x-ui:
- `tg_id`
- `tg_username` (текущий, может меняться)
- `client_email` (текущий email клиента в 3x-ui = ключ)
- `client_uuid` (`client.id`, нужен для `update`/`delete`)
- `inbound_id` (-ы), куда добавлен клиент
- `sub_id` + готовая ссылка на подписку
- `trial_issued_at` (когда выдан триал) — для отсчёта +4ч
- `trial_expiry_ms` — для отсчёта «за 3 часа до конца»
- `paid_until_ms` — для уведомления «за 1 день»
- `state` / флаги: `trial_active`, `awaiting_username`, `awaiting_receipt`, `paid`
- запланированные задачи (id джобов планировщика), чтобы их можно было отменять

> Почему отдельная БД: у 3x-ui нет нашего бизнес-состояния (этап воронки,
> ждём ли чек, какие напоминания уже отправлены). 3x-ui хранит только сам VPN-ключ.

## 1. Старт бота — `/start`

**Действие пользователя:** команда `/start`.

**Бот:**
1. Показывает приветствие (что это за VPN, что можно попробовать бесплатно).
2. Под приветствием — одна inline-кнопка **«Попробовать»** (`callback_data="trial"`).

**API:** нет (чистый Telegram).

> Если пользователь уже нажимал «Попробовать» (есть запись в БД) — можно либо
> показать его статус/ссылку, либо снова приветствие. Решить на этапе реализации;
> рекомендую: показать текущий статус + кнопку «Оплатить».

---

## 2. Нажата кнопка «Попробовать» — выдача триала

**Действие пользователя:** нажал «Попробовать».

**Бот делает:**

1. Определяет `email`:
   - есть `message.from_user.username` → `email = "@" + username`
   - нет username → `email = "temp-" + str(user_id)` (на этом этапе это ОК)
2. Проверяет, нет ли уже клиента с таким email (анти-дубликат).
   **API:** `api.client.get_by_email(email)` → если уже есть, не создаём второй.
3. Получает список всех инбаундов (триал нужно повесить на ВСЕ).
   **API:** `api.inbound.get_list()` → берём все `ib.id`.
4. Генерирует `uuid` и `sub_id`, создаёт клиента:
   - `expiry_time = int((time.time() + 1*86400) * 1000)` — +1 день (в миллисекундах!)
   - `limit_ip = 2`
   - `total_gb = 0` (безлимит по трафику)
   - `enable = True`, `tg_id = user_id`, `sub_id = uuid4().hex[:16]`
   **API:** `api.client.add(inbound_id, [client])` — вызвать для КАЖДОГО инбаунда
   (один и тот же `sub_id` → одна ссылка подписки на все инбаунды).
   *(см. рецепт 7.3 в шпаргалке)*
5. Собирает ссылку на подписку:
   `https://<host>:<subPort>/<subPath>/<sub_id>` (см. п.7.6 шпаргалки; обычно `:2096/sub/`).
6. Сохраняет в свою БД: `client_email`, `client_uuid`, `inbound_id(-ы)`, `sub_id`,
   `trial_issued_at = now`, `trial_expiry_ms`, `state = trial_active`.
7. Отправляет в чат ссылку на подписку + краткую инструкцию по подключению.
8. **Запускает сценарий отложенных уведомлений** (см. п.3).

**API итого:** `client.get_by_email`, `inbound.get_list`, `client.add`.

---

## 3. Сценарий отложенных уведомлений (после выдачи триала)

Планируем задачи в планировщике (APScheduler / собственный лупер).

### 3.1 Через 4 часа после выдачи — проверка факта использования трафика

**Бот (джоба «check_trial_4h»):**
1. Берёт актуальные данные клиента.
   **API:** `api.client.get_by_email(email)` → смотрим `up`, `down`.
   (альтернатива: пройтись по `inbound.get_list()` → `client_stats`, рецепт 7.5/7.1).
2. `used = up + down`.

- **Если `used > 0` (трафик ИСПОЛЬЗОВАЛСЯ):**
  → отправляем сообщение с предложением оформить подписку:
  «Понравилось? Подписка 200 ₽/мес» + кнопка **«Оплатить»** (`callback_data="pay"`).
  → джобу «remind_3h_before» НЕ запускаем (или отменяем).

- **Если `used == 0` (трафиком НЕ пользовались):**
  → ничего не шлём сейчас.
  → планируем джобу «remind_3h_before» на момент `trial_expiry_ms - 3*3600*1000`.

**API:** `client.get_by_email` (или `inbound.get_list`).

### 3.2 За 3 часа до конца ключа (только если триалом не пользовались)

**Бот (джоба «remind_3h_before»):**
1. Перепроверяет трафик (вдруг за это время начали пользоваться).
   **API:** `api.client.get_by_email(email)`.
   - Если `used > 0` теперь → ведём как «трафик использовался»:
     шлём оффер с кнопкой **«Оплатить»**.
   - Если всё ещё `0` → шлём напоминание-просьбу протестировать VPN
     («Остался ключ на 3 часа, попробуй подключиться»), можно тоже с кнопкой **«Оплатить»**.

**API:** `client.get_by_email`.

> Во ВСЕХ сообщениях, где предлагаем платить, кнопка одна и та же — **«Оплатить»**
> (`callback_data="pay"`). Дальше всё уходит в единый обработчик п.4.

## 4. Нажата кнопка «Оплатить» — гейт по username + оплата

Единый обработчик `callback_data="pay"`.

### 4.1 Проверка username

**Бот:**
1. Берёт текущий `message.from_user.username`.
2. Условие «username корректен» = username существует И `email` клиента в 3x-ui
   уже в формате `@username` (а не `temp-<id>`).

### 4.2 Username ОТСУТСТВУЕТ (или email ещё `temp-...`)

**Бот:**
1. Отправляет инструкцию: как поставить username в настройках Telegram.
2. Под инструкцией — кнопка **«Username установлен»** (`callback_data="username_done"`).
3. Ставит флаг `awaiting_username = True`.
4. **На оплату НЕ пускает.**

**Обработка «Username установлен» (`callback_data="username_done"`):**
1. Заново читает `message.from_user.username`.
   - Всё ещё пусто → повторяет инструкцию (username не появился), остаётся на этом шаге.
   - Появился → продолжаем:
2. Новый `email_new = "@" + username`.
3. Обновляет email клиента в 3x-ui.
   **API:** способа «переименовать email» одним вызовом в SDK нет.
   `email` — это ключ клиента, поэтому:
   - читаем клиента: `c = api.client.get_by_email(old_email)`;
   - меняем `c.email = email_new`;
   - сохраняем: `api.client.update(c.id, c)` (первый аргумент — UUID `c.id`).
   > Если на конкретной версии панели `update` не даёт сменить email —
   > запасной путь: создать клиента заново с новым email (`client.add`)
   > и удалить старого (`client.delete(inbound_id, old_uuid)`),
   > сохранив прежние `expiry_time`, `total_gb`, `sub_id`, `uuid`.
   > **Проверить на своей версии при реализации.**
4. Обновляет свою БД: `client_email = email_new`, `tg_username = username`,
   `awaiting_username = False`.
5. **Только после успешного обновления email** — повторно запускает сценарий оплаты (п.4.3).

**API:** `client.get_by_email`, `client.update` (резерв: `client.add` + `client.delete`).

### 4.3 Username КОРРЕКТЕН — сценарий оплаты

**Бот:**
1. Отправляет сообщение со статической ссылкой на оплату (200 ₽).
2. Просит прислать чек (PDF).
3. Ставит флаг `awaiting_receipt = True`.

**API:** нет (ссылка статическая, оплата вне 3x-ui).

## 5. Приём и парсинг чека → активация месяца

**Действие пользователя:** присылает PDF-файл чека (когда `awaiting_receipt = True`).

**Бот:**
1. Скачивает PDF из Telegram.
2. Парсит PDF, извлекает текст.
   - Библиотеки: `pdfplumber` / `PyPDF2` / `pypdf` (текстовый слой).
   - Если чек — картинка/скан без текста → OCR (`pytesseract`) как резерв.
3. Проверяет факт перевода на сумму **200 ₽**:
   - ищет сумму (200, «200,00», «200.00 RUB/₽/руб»);
   - (рекомендуется) ищет признак успешного перевода / реквизиты получателя,
     чтобы не зачесть чужой/левый чек;
   - (рекомендуется) защита от повторного использования одного чека:
     хранить хэш/номер операции в БД.
4. **Если чек валиден (перевод на 200 ₽ подтверждён):**
   - читает клиента: `c = api.client.get_by_email(email)`;
   - продлевает на месяц:
     `base = max(now_ms, c.expiry_time)`;
     `c.expiry_time = base + 30*86400*1000`;
     `c.enable = True`;
   - сохраняет: `api.client.update(c.id, c)` (рецепт 7.2 шпаргалки);
   - в БД: `paid = True`, `paid_until_ms = c.expiry_time`, `awaiting_receipt = False`;
   - отвечает пользователю: «Оплата принята, подписка активна до <дата>»;
   - планирует джобу «notify_1day_before» на `paid_until_ms - 1*86400*1000`.
5. **Если чек НЕ распознан / сумма не та:**
   - просит прислать корректный чек (остаётся в `awaiting_receipt`);
   - (опц.) после N неудач — уведомление админу для ручной проверки.

**API:** `client.get_by_email`, `client.update`.

### 5.1 За 1 день до конца платной подписки

**Бот (джоба «notify_1day_before»):**
1. (Желательно) перепроверяет срок.
   **API:** `api.client.get_by_email(email)` → актуальный `expiry_time`.
2. Шлёт уведомление «подписка заканчивается завтра» + кнопка **«Оплатить»**
   (снова уходит в п.4 — но username уже корректен, значит сразу п.4.3).

**API:** `client.get_by_email`.

## 6. Админ-доступ в оригинальный UI 3x-ui (Telegram WebApp)

**Идея:** для админа (по его tg user_id) бот даёт кнопку, открывающую
оригинальную веб-панель 3x-ui прямо внутри Telegram (WebApp = встроенный браузер
с быстрым входом). UI рисует сам 3x-ui, бот ничего не отрисовывает.

**Реализация:**
1. В боте захардкожен/в конфиге список `ADMIN_IDS`.
2. Команда (напр. `/admin` или `/panel`), видимая только админам.
3. Бот отправляет сообщение с inline-кнопкой типа **WebApp**:
   `InlineKeyboardButton(text="Открыть 3x-ui", web_app=WebAppInfo(url="<host>/<base-path>/panel/"))`.
4. Для «быстрого входа» варианты:
   - просто открыть URL панели (логин руками 1 раз, сессия в куках WebView);
   - либо отдавать ссылку с уже валидной сессией (если панель это поддерживает).
   > Аутентификацию панели бот не подменяет — это просто открытие её URL в WebApp.

**API 3x-ui:** напрямую не нужен (открываем родной web-UI).
Нужен только Telegram Bot API: `WebAppInfo` / inline web_app кнопка.

> Примечание: Telegram WebApp требует, чтобы URL был по HTTPS.
> Панель 3x-ui должна быть доступна по https (валидный или хотя бы рабочий TLS).

## 7. Итоговая карта «шаг → методы API»

| Шаг | Telegram-событие | Методы py3xui |
|-----|------------------|---------------|
| 1. `/start` | команда | — |
| 2. «Попробовать» | callback | `client.get_by_email`, `inbound.get_list`, `client.add` |
| 3.1 +4ч проверка | джоба | `client.get_by_email` (или `inbound.get_list`) |
| 3.2 −3ч напоминание | джоба | `client.get_by_email` |
| 4.1 «Оплатить» проверка | callback | (чтение email из БД/3x-ui) `client.get_by_email` |
| 4.2 смена username→email | callback | `client.get_by_email`, `client.update` (рез.: `add`+`delete`) |
| 4.3 выдача ссылки оплаты | — | — |
| 5. парсинг чека → месяц | документ | `client.get_by_email`, `client.update` |
| 5.1 −1 день уведомление | джоба | `client.get_by_email` |
| 6. админ-вход в UI | команда | — (Telegram WebApp) |

**Полный набор методов API, который понадобится:**
- `api.login()` — старт бота (держать один объект `AsyncApi` на всё приложение, релогин по ошибке).
- `api.inbound.get_list()` — все инбаунды (для триала на все + поиск истекающих).
- `api.client.get_by_email(email)` — статус/трафик/срок/uuid клиента.
- `api.client.add(inbound_id, [client])` — создание клиента (триал).
- `api.client.update(uuid, client)` — продление, смена email, enable.
- (резерв) `api.client.delete(inbound_id, uuid)` — для пересоздания при смене email.

**Не нужны для этого ТЗ (но есть в API):** `reset_ips`, `reset_stats`, `online`,
`delete_depleted`, `server.*`, `database.*` — пригодятся для расширений
(контроль устройств, чистка, мониторинг), но в базовый workflow не входят.

## 8. Технологический стек (рекомендация)

- **aiogram 3.x** — Telegram-бот (async).
- **py3xui (AsyncApi)** — работа с 3x-ui (см. шпаргалку, async-вариант).
- **APScheduler** — отложенные джобы (+4ч, −3ч, −1день). Джобы привязывать к `tg_id`/email,
  хранить их id, чтобы отменять (напр. отменить «−3ч», если трафик пошёл).
- **SQLite (через aiosqlite/SQLAlchemy)** — бизнес-состояние пользователей и флаги воронки.
- **pdfplumber / pypdf** (+ `pytesseract` как OCR-резерв) — парсинг чека.
- Креды 3x-ui — через переменные окружения (`XUI_HOST/USERNAME/PASSWORD`), см. шпаргалку §1.

## 9. Открытые вопросы (решить при реализации)

1. Смена `email` через `update` — работает ли на твоей версии панели, или нужен путь
   «создать заново + удалить старого»? Проверить через Swagger (`API Docs`).
2. Точный формат ссылки подписки (`subPort`, `subPath`) — взять из Settings → Subscription.
3. Валидация чека: какие именно реквизиты/получатель должны быть в PDF, чтобы зачесть оплату
   (защита от подделок и повторов).
4. Поведение `/start` для уже зарегистрированного пользователя (показывать статус?).
5. Один `sub_id` на все инбаунды — подтвердить, что подписка корректно отдаёт все ключи.
